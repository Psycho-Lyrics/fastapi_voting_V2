# Посещение домашней страницы

**Общий сценарий**
	1. Пользователь посещает домашнюю страницу
	2. Домашней страницей будет выступать **главная**
	3. Видит ленту с доступными голосованиями
	4. Различные параметры для правил отображения списка голосований
	5. Форму авторизации/регистрации, если требуется
	6. Имеет возможность ознакомиться с доступными голосованиями подробнее

## UI. Описание главной страницы
- Гость
	- Голосования
		Вертикальная, пагинированная лента с демо-блоками информации о голосовании следующего содержания:
		1. Заголовок голосования
		2. Краткое описание для посвящения в цель проведения опроса
		3. Флажок-статус: **анонс**, **регистрация**, **сбор голосов**
		4. Дата начала и завершения текущего этапа
		5. Отдел(ы), которому принадлежит голосование. **В контексте гостя флажок "Публичное"**
		6. Сам блок реагирует на нажатия **ненавязчивым** сообщением о необходимости авторизоваться
		*Примечания*:
			— Гостю, непрошедшему авторизацию, к ознакомлению доступны только публичные голосования
			— В случае, если полученный список голосований пуст, отобразить какую-то **дружелюбную надпись** без каомодзи
			— События с наведением курсора на блок голосования должны недвусмысленно демонстрировать кликабельность
	- Поиск
		Поле произвольного ввода для поиска по голосованиям, без подсказок
	- Фильтры
		К ленте с голосованиями должны причастно примыкать следующие элементы
		1. Кнопки-переключатели: "Активные" и "Архивные". Кнопка "Архивные" должна оставаться неактивной до прохождения авторизации
		2. Базовые условия фильтрации: **Статус**, **Временные условия around дат начала/завершения этапов опроса**
	- Сортировка
		Элемент(ы), причастно примыкающие к ленте с голосованиями
		1. "Сначала новые"
		2. "Сначала старые"
	- Авторизация
		1. Адрес электронной почты
		2. Пароль
		3. Строка-ссылка для восстановления доступа "Забыли пароль?"
		4. Пункт-приглашение к форме регистрации [[Технические вопросы#^fa9a53|Переход с авторизации к регистрации]]
		5. Галочка "Запомнить меня:"
		6. Кнопка-триггер для отправки данных
	- Регистрация [[Технические вопросы#^d89690|Альтернативная регистрация]]
		1. Имя
		2. Фамилия
		3. Отчество(Опционально)
		4. Адрес электронной почты
		5. Пароль
		6. "Подтвердить пароль"
		7. Пункт-приглашение к форме авторизации
		8. Возле списка с выбором отдела галочка "Глава отдела?"
		9. Кнопка-триггер для отправки данных
		*Примечания*:
			— Адрес электронной почты подвергнуть проверке формата
			— Совпадение паролей проводить на UI
			— Галочка главы отдела: подразумевается, что **сотрудник** может находится на различных ставках уникальных отделов, поэтому выбор отделов **множественный** по умолчанию. В случае с **главой отдела** считается, что одновременно можно занимать пост главы только в одном отделе. Пользователь может выбрать несколько отделов(родительский с дочерними, например) и указать себя главой, в таком случае к данным для отправки привязывается **один** самый старший в указанной иерархии отделов

## **UX**. Описание технической концепции
### Момент посещения главной страницы
При посещении Front-приложение оценивает сценарий: наличие сессионных данных, ревизия localStorage, cookie. Определяет аргументы и совершает запрос на выборку к серверу<br>
В зависимости от результата существует три сценария:
1. Гость
2. Сотрудник
3. Глава отдела <br>
Принципиально сценарии не отличаются, всё в любом случае заканчивается запросом к серверу и отгрузкой полученных данных, но выделю, что:
	- Гостю доступны только **публичные** голосования
	- Сотруднику только публичные и те, что принадлежал его отделу
	- Глава отдела видит **собственные** голосования, **доступные его отделу** и **публичные**
<p></p>
	Отдельные сущности конкретного блока голосований в ленте
		1. Заголовок - возвращает сервер
		2. Описание - возвращает сервер
		3. Флажок-статус - возвращает сервер
		4. Даты начала и завершения текущего этапа - возвращает сервер
		5. Строка с именем и фамилией создателя - возвращает сервер
		6. Отделы голосования - возвращает сервер
		7. Тег для голосований, создателем которых является текущий пользователь, определяется серверной частью. Подробнее в **body**
<p></p>
### Контракты
#### URL
`domain/api/v1/voting/get_available/?<query>`
	**query:** 
		`page:<integer>, required, def=1`
		`status:<announce|register|collecting|active|archived>, required, def=active`
		`owner:<any_string>|own, def=None`
		`tamp_field:created|announce|register|collecting|active|archived, def=None`
		`tamp_dir:until|after|at, def=None`
		`tamp_date:<date>, def=None`
	**Примечание**
		— Параметры со значением пары **required** и **def(default)** подразумевает обязательство фронт-части включать аргументы в запрос со значением по умолчанию. Наличие только **def(default)** констатирует поведение сервера в случае, если необязательный аргумент не был передан
		— **owner** принимает два аргумента: первый подразумевает поиск по создателям. Это предусматривают все сценарии. Второй аргумент определяет фильтр для выборки собственных голосований в ответ на запрос главы отдела
		— **tamp-параметры**. Трио параметров, в совокупности определяющих единое условие выборки - **до/после конкретного этапа голосования**. Наличие одного аргумента определяет обязательство указать оставшиеся, проверки - будут.
		— **Фильтрация по отделам** определяется аргументами в **body**! Читаем дальше
#### Headers
**Request:**
- `access_token:<token>`
	**Примечание**
		— Конечно, токен **не** требуется, если пользователь не авторизован(сценарий гостя)
**Response:**
В первой версии заголовков, кроме системных, в ответе не ожидается

#### Body
**Request**:
```json
{
	deps[<uuid>]
}
```
*Примечание*
	Формат аргумента для запроса с фильтрацией по отделам

**Response**:
```json
{
votings:[
	{
	uuid:UUID, 
	name:string, 
	about:string,

	expire_status:date,
	owner:string,
	voting_deps:{
		uuid:uuid,
		name:string
		}
	}
],
pagination:{
	cur_page:int,
	page_count:int,
	has_next:bool,
	has_back:bool
	}
}
```
